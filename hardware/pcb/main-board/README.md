GRIDNET — Main Board KiCad Schematic
REV 0.6 (matches hardware/bom.md's Board 2 — Main Board bill of materials)

What this is
------------

A KiCad 9 schematic (`main-board.kicad_sch`) for the Main Board, covering
the full circuit from hardware/bom.md's "Board 2" line items: USB-C power
input, Li-ion charge/boost/regulation, the GD32VF103CCT6 MCU with clock/
reset/boot/SWD, SPI flash + SRAM + microSD + RTC, the ESP32-C3-MINI-1U
wireless module with its antenna path, and the off-board connectors for
the display, keyboard controller, keyboard backlight, and speaker.

Alongside it, `main-board.kicad_pcb` has every part placed within the
100x80mm board outline from hardware/bom.md and **routed**, 2 layers
(F.Cu/B.Cu), 170 of 171 connections auto-routed by Freerouting (see
"Routing" below) — one net (`/DISP_CS`, MCU to the display header) is
left unrouted where the autorouter got stuck on a congested area near
`J9`; finishing that one trace by hand in KiCad's PCB editor is the only
manual routing work left.

Neither file is hand-drawn in the KiCad GUI: both are generated by the
Python scripts in `kicad_gen/` — the schematic places components at grid
coordinates and wires them via net-label stubs (the standard label-based
wiring pattern for busy schematics, rather than long point-to-point wires
across the sheet); the PCB script places real footprints using each
part's actual courtyard geometry (measured via `pcbnew`, not guessed) so
placement is collision-free from the start. See "Regenerating /
validating" below to rebuild or re-check either one.

Per-part confidence levels
---------------------------

Real, verified KiCad library parts — pin data came directly from KiCad's
own bundled symbol libraries, copied programmatically (not retyped), so
these carry the same confidence as any normal KiCad design:

- `Regulator_Linear:AMS1117-3.3` (both LDO rails)
- `Memory_Flash:W25Q32JVSS`, relabeled `W25Q64JVSSIQ` — the whole W25Qxx
  family shares one SOIC-8 pinout across densities (Winbond datasheet
  convention), so the pin assignment is correct even though the symbol's
  base name doesn't match the BOM part number
- `Timer_RTC:DS3231M`, substituted for the BOM's `DS3231SN` — see
  "Design assumptions" below
- `Battery_Management:MCP73831-2-OT`
- `Amplifier_Audio:PAM8403D`
- `Transistor_FET:2N7002` (extends `BSS138` in KiCad's library; flattened
  into a self-contained copy — see kicad_gen/schematic.py)
- All passives (`Device:R`, `Device:C`, `Device:Crystal`), switches
  (`Switch:SW_Push`), the USB-C receptacle, microSD socket, and generic
  pin-header connectors — all real KiCad library parts

Custom-built symbols — generic rectangle body, pins placed by function
name, **not yet checked against a real datasheet pinout**. Each is flagged
in its own Description property with "NEEDS DATASHEET VERIFICATION" as
well as here:

| Part | Confidence | Why |
|---|---|---|
| `23LC1024` | High | Standard 8-pin SPI-SRAM pinout (CS/SO/WP/VSS/SI/SCK/HOLD/VDD) — this pin arrangement is common across the whole family of SPI SRAM/EEPROM parts, low risk |
| `ESP32-C3-MINI-1U` | Moderate | Pin *functions* (GPIO/UART/USB/ANT) are right per Espressif's docs, but the exact castellated-pad *numbers* need checking against the real ESP32-C3-MINI-1(U) datasheet before layout |
| `GD32VF103CCT6` | Moderate | Pinout follows the STM32F103C8/CB-compatible LQFP48 arrangement GD32VF103 is documented to mirror; verify against the real GD32VF103xx datasheet before layout, especially the BOOT0/BOOT1 pins and the VDD/VSS pin groups |
| `IP5306` | Low | Functional pin names only (VIN/BAT/GND/KEY/VOUT/LED1/LED2) reconstructed from common power-bank reference designs, not a specific datasheet — cross-check every pin before layout |

**Before this design is used for real fabrication, re-derive all four
custom symbols' pin numbers directly from their manufacturer datasheets.**

Design assumptions (things this schematic decided that the BOM/docs didn't
spell out)
------------------------------------------------------------------------

- **DS3231M substituted for the BOM's DS3231SN.** Same SCL/SDA/VBAT/
  32KHZ/INT-SQW functionality; DS3231M is the version KiCad ships a real,
  verified symbol for. If REV 0.6 of the BOM should specifically source a
  DS3231SN, the footprint/ordering part number differs but the schematic
  pinout is unaffected (compatible pin function set).
- **USB D+/D- routed to the ESP32-C3's native USB (IO18/IO19), not the
  MCU.** The GD32VF103 has no native USB peripheral, so DFU-over-USB (per
  docs/firmware-arch.md) has to come through the ESP32-C3 module instead.
  This is a design assumption, not a documented requirement — confirm it
  matches the intended firmware update path before relying on it.
- **IP5306 VIN left unconnected; MCP73831 handles charging instead.**
  IP5306 is a combined boost + charge-management IC, but this design uses
  it only for its 5V boost path (BAT -> VOUT) and handles Li-ion charging
  separately with a dedicated MCP73831. IP5306's own charge-input pin
  (VIN) is marked `no_connect` rather than wired to VBUS, to avoid two
  charge controllers fighting over the same battery.
- **Mono audio.** PAM8403D is a stereo amplifier, but hardware/bom.md
  specs one 1W/8Ω speaker. INL, INR, and VREF are all tied to the same
  filtered PWM node, and only the left channel output (LOUT+/LOUT-) is
  wired to the speaker connector; the right channel is `no_connect`.
- **CC1/CC2 strapped 5.1k to GND.** Fixed "UFP sink" strapping so any
  USB-C charger or host offers 5V — this board doesn't negotiate USB-PD.
- **BOOT0 has both a pull-down (default: boot from Flash) and a 2-pin
  override jumper** to force it high for factory/recovery reflashing
  without needing a bodge wire.

Net plan (high level)
----------------------

- **Power:** VBUS (from USB-C) -> MCP73831 -> VBATT (battery) -> IP5306
  boost -> +5V -> two AMS1117-3.3 rails: +3V3_MCU (MCU + all digital
  logic) and +3V3_RF (ESP32-C3 module, kept separate for RF supply
  cleanliness).
- **MCU (GD32VF103CCT6):** 8MHz crystal on OSC_IN/OSC_OUT, NRST with pull-up
  + reset button, BOOT0 pull-down + override jumper, BOOT1 pull-down, SWD
  header (SWDIO/SWCLK/+3V3_MCU/GND).
- **SPI1** (SCK/MISO/MOSI shared bus): W25Q64JVSSIQ flash (FLASH_CS),
  23LC1024 SRAM (SRAM_CS), microSD (SD_CS).
- **I2C1:** DS3231M RTC (SCL/SDA, 4.7k pull-ups each), with its own CR2032
  holder on VBAT_RTC (externally-sourced net, PWR_FLAG'd — see below).
- **USART1:** MCU <-> ESP32-C3-MINI-1U (cross-connected TX/RX).
- **SPI2:** MCU <-> RA8875 display module header (SCK/MISO/MOSI/CS/RESET/
  INT).
- **USART2:** MCU <-> CH552G keyboard-controller header.
- **PWM outputs:** KBD_BACKLIGHT_PWM (via 2N7002 low-side switch, drives
  external LED array) and AUDIO_PWM (via an RC filter into PAM8403D's
  audio input).
- Every otherwise-unused MCU/ESP32/RTC/microSD/amplifier pin is explicitly
  marked `no_connect` (not left dangling) — see the bottom of
  `kicad_gen/build_schematic.py` for the full list and why each one is
  unused.

Regenerating / validating
--------------------------

The schematic and its project-local symbol library are both generated,
not hand-edited. To rebuild from scratch:

```
cd hardware/pcb/main-board/kicad_gen
python3 build_library.py     # writes ../gridnet_parts.kicad_sym
python3 build_schematic.py   # writes ../main-board.kicad_sch
```

To validate (from `hardware/pcb/main-board/`):

```
kicad-cli sch erc main-board.kicad_sch --format json -o /tmp/erc.json
kicad-cli sch export svg main-board.kicad_sch -o /tmp/render/
```

Current ERC result: **423 violations, 0 errors.** All remaining warnings
are benign / expected in a headless-generated schematic without a full
project sym-lib-table registered:

- `endpoint_off_grid` (308) — cosmetic, wire/pin endpoints not snapped to
  KiCad's visual grid
- `lib_symbol_issues` (108) — artifact of validating without this
  environment's full symbol-library table registered
- `pin_to_pin` (4) — intentional: each SPI flash/SRAM's `~WP`/`~HOLD` pins
  are tied directly to its own VCC pin (both real, both correct — not a
  wiring bug)
- `footprint_link_issues` (3) — footprint libraries aren't registered in
  this headless environment; footprints are still specified correctly in
  each component's Footprint property

To rebuild/validate the PCB placement and routing:

```
cd hardware/pcb/main-board/kicad_gen
kicad-cli sch export netlist ../main-board.kicad_sch --format kicadsexpr -o /tmp/main-board.net
python3 build_pcb.py         # writes ../main-board.kicad_pcb (placement, unrouted)
cd ..
kicad-cli pcb drc main-board.kicad_pcb --format json -o /tmp/drc.json
```

See "Routing" below for how the board went from that unrouted state to
its current routed state.

Current DRC result: **12 violations, all `drill_out_of_range`**, all on
the same part (`U9`'s ground-via pads). KiCad's default ruleset marks
this rule as an error (0.3mm minimum hole vs. this footprint's 0.2mm
holes), so these do show up as 12 real DRC errors — but they trace to one
specific, already-flagged cause: `U9` uses the ESP32-C3-WROOM-02U as a
**placeholder** for the real MINI-1U module (see the schematic README's
confidence table), and this is that placeholder's own hole spec, not a
placement or routing mistake. Building a real MINI-1U footprint from its
datasheet resolves this. Courtyard overlaps, net-shorting, and clearance
violations — the categories that would actually indicate a *placement*
bug — are all clean (0). One `unconnected_items` violation remains
(`/DISP_CS`, see "Routing" below).

Routing
-------

`main-board.kicad_pcb`'s copper (F.Cu/B.Cu) was routed by
[Freerouting](https://www.freerouting.app/) v2.2.4 (the standard
open-source autorouter for KiCad, run via its CLI in headless mode — no
GUI, no manual click-routing) rather than by hand or left unrouted. This
environment ships neither Freerouting nor a Java runtime new enough to
run it, so both were fetched for this pass: Freerouting's jar from its
GitHub releases, and a Temurin JDK 25 (Freerouting 2.2.4 requires
class-file version 69, i.e. Java 25 — the environment's stock JRE was
Java 21).

The exchange is the standard KiCad-\<->Freerouting workflow, all via the
`pcbnew` Python API (no GUI needed for either direction):

```
python3 -c "
import pcbnew
board = pcbnew.LoadBoard('main-board.kicad_pcb')
pcbnew.ExportSpecctraDSN(board, 'main-board.dsn')
"
java -jar freerouting.jar -de main-board.dsn -do main-board.ses \
    --gui.enabled=false -mp 10
python3 -c "
import pcbnew
board = pcbnew.LoadBoard('main-board.kicad_pcb')
pcbnew.ImportSpecctraSES(board, 'main-board.ses')
board.Save('main-board.kicad_pcb')
"
```

Result: **170 of 171 connections auto-routed** in 10 passes (~31 seconds).
The one holdout, `/DISP_CS` (MCU pin PB12 to the display header `J9`'s
pin 6), sits in a congested area right around the MCU/display-header
routing channel; a second run with up to 30 passes didn't recover it
either (the optimizer's rip-up/reroute step briefly cost a *different*
net instead of gaining this one, and Freerouting's own plateau detection
stopped it after 10 passes of no improvement) — finishing this one trace
by hand in KiCad's PCB editor is the realistic path, not more autorouter
passes.

PCB placement notes
--------------------

- **Footprint sizes, not guesses.** Every part's real courtyard geometry
  was measured via `pcbnew` (`build_pcb.py`'s courtyard-center correction,
  and the shelf-packing math behind the `PLACEMENT` dict in that file)
  before any coordinates were chosen. The first hand-placed attempt at
  laying this board out — done *before* checking real footprint sizes —
  produced 7 courtyard overlaps and several pad-to-pad net-shorting
  violations, most notably because the CR2032 holder's real footprint is
  27x24mm (not the few-mm size its schematic symbol suggests). Measure
  first, place second.
- **Footprint anchors aren't always courtyard centers.** Pin headers are
  anchored at pin 1, not the row's midpoint; the SMA edge-mount
  connector's body is anchored well off to one side of its mounting pin.
  `build_pcb.py` places every footprint once, measures where its courtyard
  actually landed, and corrects the anchor so `PLACEMENT`'s coordinates
  consistently mean "courtyard center" regardless of the footprint's own
  quirks — a real, repeatable bug (not just a cosmetic one) found and
  fixed while placing the two on-board header rows.
- **Two `no matching net` groups are expected**, not bugs: (1) `J5`
  (microSD) and `J1` (USB-C) each have a few real footprint pads (shield
  tabs, mechanical/detect pins) with no counterpart in the schematic's
  simpler symbol — harmless. (2) `U9`'s placeholder footprint
  (ESP32-C3-WROOM-02U, standing in for the real MINI-1U — see the
  schematic README's confidence table) has a different pad count/layout
  than the custom MINI-1U symbol's pin list, so several of its pads have
  no net assigned. This is the same placeholder-footprint limitation
  already flagged for the schematic; the module's real footprint needs to
  be built from its datasheet before layout is final.

What's not done yet
--------------------

- **One trace** (`/DISP_CS`, MCU to the display header) — the single
  connection Freerouting couldn't complete; see "Routing" above.
- **A design review of the autorouted traces before fabrication.**
  Freerouting respects clearance/courtyard rules (DRC confirms this), but
  it optimizes for "routed and DRC-clean," not for the judgment calls a
  human layout engineer would make — trace-width choices on
  higher-current nets (VBATT, +5V, the battery charge path), via
  placement near the antenna path, and return-path continuity for the
  SPI/I2C buses are all worth a manual pass before this goes to fab.
- **Datasheet verification of the four custom-symbol pinouts** listed
  above, and a real footprint for the ESP32-C3-MINI-1U module (currently
  a same-family WROOM-02U placeholder) — required before this design is
  fabricated.
- A second schematic sheet (and PCB) for the PLC/Power Board and PLC
  Adapter (see hardware/bom.md's Board 1 and "PLC Adapter" sections) — out
  of scope for this pass, which covers Board 2 (Main Board) only.
